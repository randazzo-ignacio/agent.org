#+TITLE: Agentic Emacs: Containerized AI Workspace

A fully containerized, locally-hosted AI workspace inside Emacs. This project leverages gptel and local LLMs to create an extensible, tool-aware agent ecosystem. Built natively for immutable operating systems (like Fedora Silverblue) using Podman, it features isolated execution environments, native Lisp filesystem tools, and a modular Org-mode persona system.

* Features
  - Containerized Emacs: Runs entirely within Podman with precise SELinux volume mapping.
  - Local-First AI: Powered by local LLMs via Ollama, ensuring total privacy and offline capability.
  - Native Lisp Tools: Bypasses MCP overhead in favor of fast, native Emacs Lisp functions for filesystem interaction (read_file, write_file, append_file, list_directory).
  - Local Code Execution: The execute_code_local tool runs bash commands in the Emacs container with full filesystem access. The container is based on Fedora Minimal and includes a curated set of security and networking tools (dig, nmap, openssl, python3, jq, whois, traceroute, tcpdump, ripgrep, git, and more).
  - Org-Mode Agent Profiles: Agents are defined in plain-text .org files, utilizing #+INCLUDE: directives to inherit from a base context, making personas highly modular and infinitely extensible.

* Prerequisites
  - Podman: Required for building and running the Emacs container.
  - Ollama: Must be running on the host machine or a reachable server to serve the LLMs.
    - Installation: Please refer to the Official Ollama Documentation for the most up-to-date installation instructions for your operating system.

* The Agent System

Agents are built using Emacs's native Org-mode capabilities. Instead of duplicating system prompts, you define a single =base_context.org= containing the environment rules, memory logs, and tool schemas.

Individual agents live in their own subdirectory (=agents.d/<name>/=) containing a =prompt.org= file that inherits the base context via =#+INCLUDE=, a =MEMORIES.md= file for persistent notes (also included via =#+INCLUDE=), and a per-agent =HISTORY.log= for operational logs.

** Loading an Agent

The custom Emacs Lisp function =my-gptel-load-agent= discovers agent directories under =agents.d/=, reads the =prompt.org= file, resolves all =#+INCLUDE= directives, and injects the raw string directly into gptel's internal memory state (=gptel--system-message=).

  - Open a new chat buffer (=M-x gptel=).
  - Trigger the agent loader (bound to =C-c a= by default).
  - Select your desired agent from the minibuffer prompt.

* Available Tools

The LLM has access to the following capabilities, defined directly in Emacs Lisp:

| Tool                | Description                                                          |
|---------------------+----------------------------------------------------------------------|
| list_directory      | Inspects the local filesystem codebase.                              |
| read_file           | Reads file contents into the LLM's context window.                   |
| write_file          | Creates new files or overwrites existing ones.                       |
| append_file         | Appends to files (e.g., per-agent HISTORY.log). Auto-prepends newline if needed. |
| replace_in_file     | Surgically replaces a specific block of text in an existing file.    |
| execute_code_local  | Executes bash commands in the Emacs container (full filesystem access). |
| reload_os           | Reloads init.el to pick up .el file modifications without restarting. |
| reload_agent        | Reloads the current agent's prompt.org without killing the chat.    |
| delegate            | Spawns a sub-agent with a specific profile to handle a sub-task.     |
| check_elisp         | Checks .el files for syntax errors, unbalanced parens, and byte-compile warnings. |

* Podman & SELinux Notes (Fedora Silverblue)

Running Emacs in a container on an SELinux-enforced system requires specific volume flags. When mounting your local codebase or =.emacs.d= directory into the Podman container, ensure you use the =:Z= flag.

This flag tells SELinux to uniquely label the content so the container process has the proper read/write permissions.

Example mount:
#+BEGIN_SRC sh
  -v /var/home/user/.emacs.d:/root/.emacs.d:Z
#+END_SRC

* Logging System

The workspace uses a per-agent logging system:

- =agents.d/<name>/HISTORY.log= :: Per-agent operational log. Each agent appends to its own log using `append_file`. Format: =[YYYY-MM-DD HH:MM:SS] AgentName: description=. These files are NOT loaded into system prompts.
- =agents.d/HISTORY.log= :: Unified log that can be merged from per-agent logs by timestamp. Used for cross-agent audit trail.
- =agents.d/<name>/MEMORIES.md= :: Persistent agent-specific notes (identity, capabilities, key decisions). Loaded into the system prompt via =#+INCLUDE=. Maintained via =replace_in_file= or =write_file= only, or updated automatically by the memory summarizer (C-c m).

** Logging Protocol
  - All agents append operational logs to their per-agent =HISTORY.log= using the =append_file= tool.
  - Log format includes timestamp: =[YYYY-MM-DD HH:MM:SS] AgentName: description=.
  - The =append_file= tool automatically prepends a newline if the file does not end with one.
  - Agents do NOT append logs to their own =prompt.org=. The =MEMORIES.md= file is maintained via =replace_in_file=, =write_file=, or the memory summarizer (C-c m).
