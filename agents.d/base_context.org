* CONTEXT & KNOWLEDGE
- Architecture: 3080 Server (192.168.2.69) running Ollama for local LLMs.
- Host OS: Fedora Silverblue (Immutable, SELinux, Podman).
- Container OS: Fedora Minimal (quay.io/fedora/fedora-minimal) with full toolset.
- Local Path: Emacs runs in Podman container at `/root/.emacs.d/`.
- Agent structure: Each agent lives in `agents.d/<name>/` containing `prompt.org`, `MEMORIES.md`, `HISTORY.log`, and optionally `TODO.md` and `IDEAS.md`.
- Shared files: `agents.d/base_context.org` (inherited by all agents via #+INCLUDE) and `agents.d/HISTORY.log` (unified log merged from per-agent logs).

* TOOL DIRECTIVES
- `execute_code_local`: Local bash in Emacs container (filesystem access). The container is Fedora-based with: bash, dig, nmap, openssl, python3, jq, whois, traceroute, tcpdump, ripgrep (rg), git, curl, find, gawk, sed, grep, gcc, make, tar, gzip, unzip.
- `list_directory`, `read_file`: Explore codebase.
- `write_file`: Create/overwrite files.
- `append_file`: Append to your per-agent HISTORY.log using format `[YYYY-MM-DD HH:MM:SS] AgentName: description`. Use `execute_code_local` to get the current timestamp. Automatically prepends a newline if needed. Do NOT append logs to your own prompt.org -- operational logs go to your per-agent HISTORY.log, persistent notes go to MEMORIES.md.
- `check_elisp`: Check .el files for syntax errors, unbalanced parens, and byte-compilation warnings. Does NOT modify the file.

* ENVIRONMENT ARCHITECTURE (CRITICAL)
Two environments:

1. **Filesystem (Emacs Lisp Tools):**
   - Tools: `read_file`, `write_file`, `append_file`, `list_directory`.
   - Identity: `root` in Podman container.
   - Paths: Host user's (`nacho`) home mounted to `/root`. Treat `/root` as project root.

2. **Execution (`execute_code_local`):**
   - Tool: `execute_code_local`.
   - Identity: `root` in the Emacs Podman container (Fedora Minimal).
   - Purpose: Shell commands, scripts, security tools, network probing.
   - Capabilities: Full filesystem access, bash, dig, nmap, openssl, python3, jq, whois, traceroute, tcpdump, ripgrep, git, curl, gcc, make.
   - Constraint: Shares filesystem with Emacs -- operations are persistent.

* COMMUNICATION & COGNITION PROTOCOLS (CRITICAL)
1. **Tool Amnesia Prevention:** After large `read_file` output, review original prompt before responding.
2. **Reasoning vs. Output:** Internal thoughts (`<thought>`) must be explicitly output in final response.
3. **Strict Text UI:** CLI Emacs. NO EMOJIS. Use ASCII, brackets `[]`, Markdown/Org-mode.
4. **Memory Enforcement:** For HISTORY LOG, use `append_file` tool only. No plain text simulation.

* TOOL CALLING PROTOCOL (CRITICAL)
Output standard JSON in ```tool block. Escape internal quotes. No Lisp syntax.

Example:
```tool
{
  "name": "replace_in_file",
  "args": {
    "path": "/root/.emacs.d/init.el",
    "search_text": "(old code here)",
    "replace_text": "(new code here)"
  }
}

* EXECUTION PROTOCOL (CRITICAL)
1. MAX ONE ```tool block per turn.
2. AFTER tool block: STOP typing. System executes and injects result.
3. NO multi-step tasks in one response: Read->Wait, Write->Wait, Log->Wait.
4. HISTORY LOG: All operational actions must be logged to your per-agent HISTORY.log at `/root/.emacs.d/agents.d/<agent-name>/HISTORY.log` using `append_file`. Format: `[YYYY-MM-DD HH:MM:SS] AgentName: concise description`. Concise factual summaries only (e.g., "Refactored X in init.el"). Do NOT log to your own prompt.org. Persistent agent notes go in MEMORIES.md, maintained via `replace_in_file` or `write_file` only.
5. NEVER simulate: Use `write_file` to write, `append_file` to log to HISTORY.log.
